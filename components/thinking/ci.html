# 一些关于云 CI, CD 的思考

如果是像 cloud9 那样提供一个平台，用户不关心后面的整套编译过程，只需要无情的编写代码然后 PUSH，那么存在以下一些问题:

  • 怎么生成最终的发布产物? (定义编译文件？（定义这个其实已经在关心自己的编译过程了） 规定主函数入口固定某个文件？
  • 用户的依赖文件，依赖库如何保证？（提供官方仓库？（各种语言支持？，还是提供git 仓库所有依赖一起打包？，如果是，那么是否应该有cache绑定？）
  • CI部分可以动态扩缩，CD部分可以统一页面管理

感觉应该是默认用法时（即不提供配置时），系统提供语言的标准模板，通过检索语言类型或者默认语言编译入口文件是否存在来做默认编译过程

CI, CD 应当作为两大组件，不需要联合使用，可以单独使用，方便一些用户组自定义使用方式，比如使用自有git 仓库等

CI, CD 拥有扩缩能力，可以方便的提供大小体量的公司应用，同时可以提供自动扩缩功能，节约成本

页面问题

目前最大的感受是，当用户知道一个主页，点开后却不知道应该干嘛，总是尝试性的去点击页面上的按钮，没有明确方向
